# MEMENTO — 進化の憲法

このファイルは MEMENTO の**目的・原則・変更の作法**を定める。
将来この app に手を入れる者（人間であれ Claude であれ）は、コードより先にこれを読むこと。
コードは手段であり、いくらでも書き換えてよい。ここに書かれた目的だけが不変である。

## 目的

MEMENTO は情報を届ける app ではない。**認識を届ける装置**である。
届けるべき認識は次の 7 つ。

| # | 認識 | 装置 | 状態 |
|---|------|------|------|
| 1 | 有限性 — 時間は残量のある資源 | 人生の暦・残数カウンタ | **v1 で実装** |
| 2 | 転機の事後性 — 分岐点は後からしか分からない | 一行日記・「あの年の今日」 | **v1 で実装** |
| 3 | 複利性 — 小さな反復は指数的に効く | 習慣の累積量と長期投影 | 未実装 |
| 4 | 偶然性 — 機会は接触回数に比例する | 新しい接触の週次集計 | 未実装 |
| 5 | 不可逆性 — 戻せない決断だけ慎重に | 決断ジャーナル | 未実装 |
| 6 | 解釈の分離 — 感情は事象でなく解釈から生まれる | 事実欄と解釈欄の分離 | 未実装 |
| 7 | 自己の無常 — いまの価値観も最終形ではない | 四半期の価値観スナップショット | 未実装 |

未実装の 5 つはロードマップではなく**候補**である。実装の順序と要否は
利用者の実際の変化（進化レビューで得る）だけが決める。使われない装置を
足すことは、この app にとって後退である。

## 設計原則

1. **一般論ではなく、利用者自身のデータで語る。**
   「人生は短い」ではなく「父と会えるのは推定あと 30 回」。
2. **高頻度・低強度ではなく、低頻度・高強度。**
   毎日の通知は 3 日で壁紙になる。慣れがこの app の唯一の敵である。
   通知機能は原則として持たない。開いたときに一撃で伝わることに全てを賭ける。
3. **入力の摩擦は 1 日 1 行以内。**
   続かない記録装置は存在しないのと同じ。新機能が日々の入力を増やすなら、
   その機能は原則違反を疑うこと。

### 見た目の掟

- 墨色の地（#0A0A0C 系）・明朝の数字・髪の毛ほどの罫線。飾りは金（#CFA45C）の一点のみ。
- 画面に置いてよいのは「いま認識させたいもの」だけ。説明文・バッジ・空の状態の
  埋め草を足す前に、消せるものがないか探すこと。
- 追加した要素の数だけ、既存の要素を消すことを検討する。引き算が既定、足し算が例外。

## 構造（v1）

- `memento/index.html` + `memento.css` + `memento.js` — ビルドなし・依存なしの静的ページ。
  GitHub Pages で `/memento/` として配信される。
- 認証はハブと同じ `assets/github.js`（`GH`）を同一オリジンで再利用する。
  PAT・対象リポジトリの設定はハブの接続設定と共有され、MEMENTO 側では何も尋ねない。
- データは localStorage `memento.data` を正とし、PAT があれば private の
  life-content `.web/memento.json` へ 2 秒デバウンスで書き戻す（読みは起動時のみ）。

### データ（version 1）

```json
{
  "version": 1,
  "birth": "1985-04-01",
  "horizon": 80,
  "people": [
    { "id": "p...", "name": "父", "birthYear": 1958, "perYear": 6, "until": 85 }
  ],
  "diary": { "2026-07-24": "一行" },
  "created": "2026-07-24",
  "updated": "ISO 8601"
}
```

- 統合則: スカラーと `people` は `updated` の新しい側が丸ごと勝つ。
  `diary` だけは和集合（どの端末の一行も失わない）。
  日記の削除は空文字の墓標として残す（統合で復活させないため）。
- **スキーマを変えるときは `version` を上げ、`memento.js` の `migrate()` に
  旧版からの変換を必ず書くこと。利用者のデータを 1 行たりとも失ってはならない。**

## 進化のプロトコル

この app は完成しない。利用者の変化と時代の変化に合わせて変わり続けることが仕様である。

1. `.github/workflows/memento-evolution.yml` が四半期の初日に
   「MEMENTO 進化レビュー」Issue を自動で起票する。
2. 利用者は Issue の 3 つの問いに短く答える（答えなくてもよい。空欄も情報である）。
3. 利用者がその Issue を Claude に渡す（claude.ai/code でこのリポジトリを開き、
   Issue 番号を伝えて「進化レビューを実施して」と言うだけでよい）。
4. Claude は次の順で動く。
   - まずこのファイルを読み、次に Issue の回答を読む。
   - 変更は 1 回のレビューにつき **最大 3 件**。大きく変えたい誘惑は分割で抑える。
   - 「使われていない装置の除去・縮小」を「新装置の追加」より先に検討する。
   - 実装し、下の変更履歴に 1 行を追記し、`version` 表示（画面右上のローマ数字）を
     スキーマが変わった場合にのみ上げる。
   - 変更の意図を Issue に短く報告して閉じる。

### 変えてよいもの・いけないもの

- 変えてよい: 装置の構成、画面、データスキーマ（migrate を伴う場合）、このファイルの
  「候補」と「見た目の掟」の細部、レビューの頻度や問い。
- 変えるときに一段の慎重さが要るもの: 3 つの設計原則。変えるなら理由をこのファイルに残す。
- 変えてはいけないもの: 目的（7 つの認識を届けること）。利用者のデータの互換性。
  GitHub と Claude だけで完結するという制約（外部サービス・外部 CDN を持ち込まない）。
